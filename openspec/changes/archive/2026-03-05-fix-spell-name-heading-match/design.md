## Context

`parse_spell_data` 函式負責從 Google Doc 文字中定位並解析特定法術的屬性區塊。現行邏輯使用「第一個包含目標法術英文名稱的行」作為區塊起點。但同一份文件中，其他法術的描述文字可能也會引用該法術名稱（如 Air Walk 描述中提及 `antimagic field`），導致起點定位錯誤。

## Goals / Non-Goals

**Goals:**
- 法術名稱搜尋僅匹配標題行格式（如 `反魔場（Antimagic Field）`），消除描述文字引用造成的誤判
- 找不到標題行時直接放棄解析，回傳空結果

**Non-Goals:**
- 不改變解析各欄位（成分、施法時間等）的邏輯
- 不改變 `filter_rows_for_processing` 或其他函式的行為

## Decisions

### 決策 1：使用正規表達式匹配標題行格式

標題行的特徵為：短文字 + 半形/全形括號包裹的英文名稱，且行尾無其他內容。
使用 regex `^[^:：]{0,20}[(（].*[)）]\s*$` 來識別標題行。

**替代方案**：使用更嚴格的 regex 僅匹配「中文名（English Name）」格式。
**選擇理由**：現有 heading_pattern 已有類似邏輯，保持一致性；`{0,20}` 限制前方字元數可避免匹配到長描述行。

### 決策 2：找不到標題行匹配時直接放棄

使用 Python `for...else` 語法，當迴圈未 break（即未找到匹配）時直接 `return parsed_data`（空 dict）。

**替代方案**：退而求其次匹配任意包含名稱的行。
**選擇理由**：錯誤匹配比不匹配更糟——會寫入錯誤資料到試算表。

## Risks / Trade-offs

- **風險**：若某份 Google Doc 中法術標題行格式不符合 regex 預期（如缺少括號），該法術會被跳過
  - **緩解**：此格式在所有已知文件中一致；被跳過的法術可在下次批次中重新處理
