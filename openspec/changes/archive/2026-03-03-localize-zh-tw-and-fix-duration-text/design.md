## Context

`app.py` 目前從 Google Doc 擷取法術文字後，沒有任何轉換直接寫入試算表。由於原始文件多為簡體中文，造成試算表欄位出現簡體字。另外，`[持續時間]` 的比對邏輯使用「每等級」，但遊戲術語已改為「每CL」（CL = Caster Level），導致部分持續時間無法正確識別。

## Goals / Non-Goals

**Goals:**
- 擷取資料後，在寫入前統一轉為繁體中文。
- 將 `[持續時間]` 的字串判斷由「每等級」修改為「每CL」。

**Non-Goals:**
- 不對 Google Doc 原始內容做任何修改。
- 不對已存在試算表中的歷史資料進行回溯轉換。
- 不引入新的外部 API 或雲端服務。

## Decisions

### 決策 1：簡繁轉換函式庫選擇 — 使用 `opencc-python-reimplemented`

- 採用 `opencc-python-reimplemented` 套件（純 Python 實作，無外部 C 依賴）。
- 轉換設定選用 `s2twp.json`（簡體→臺灣繁體，含詞彙調整）。
- 替代方案：`zhconv`（簡單但轉換品質較低）、`opencc`（需 C 編譯環境，Windows 不易安裝）。

### 決策 2：轉換時機 — 在欄位值寫入前套用

- 在每個欄位值被賦值至更新字典前，先套用繁體轉換函式。
- 這樣可確保所有欄位（含 `[持續時間]`、`[施法時間]`、`[施法距離]` 等）都已轉換。
- 轉換後再執行字串比對（如「每CL」），避免來源仍為簡體時比對失敗。

### 決策 3：`[持續時間]` 字串處理邏輯

- **步驟 1（替換）**：將文字中的「每等級」取代為「每CL」（在長度判斷之前執行）。
- **步驟 2（截斷）**：替換後若文字長度 > 16 字，回傳「其他」；否則保留替換後的文字。
- 此替換在繁體轉換完成後進行，確保來源文字已是繁體，再執行字串比對與替換。

## Risks / Trade-offs

- **套件相依性**：新增 `opencc-python-reimplemented` 依賴 → 需更新 `pyproject.toml`（低風險）。
- **轉換準確性**：極少數專有名詞可能轉換不如預期 → 可接受，因原始文件品質已先行決定結果。
- **歷史資料不受影響**：只針對本次執行範圍的資料做轉換，舊資料不處理（此為 Non-Goal）。
